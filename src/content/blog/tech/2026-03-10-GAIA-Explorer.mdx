## 🚀 High-Level Vision

An exploration crew travels through a procedurally dramatized but scientifically grounded Milky Way.
The star map is generated from Gaia DR3/DR4 data: real positions, magnitudes, temperatures, classifications, and distances.
The player must navigate, avoid hazards, manage ship systems, and resupply at rare safe harbors.

---

## 🌌 1. Core Architecture Overview

### 1.1 Engine Layer (Metal + Swift) 🚀

The engine layer provides rendering, simulation timing, and platform integration on macOS.
It is responsible for turning Gaia-derived spatial data into a navigable 3D environment.

**Responsibilities**

- Create and manage the Metal device, command queue, and render pipelines.
- Maintain a stable simulation loop (fixed timestep for logic, variable for rendering).
- Render large starfields efficiently using GPU instancing and culling.
- Maintain a floating origin to avoid precision loss when working with large coordinate values.

**How to implement it**

- Use Swift for game logic and Metal for GPU work.
- Build two primary render pipelines:
  - A starfield pipeline for point-like or billboarded stars.
  - A geometry pipeline for ships, stations, and UI overlays.
- Use compute shaders for:
  - Frustum culling of stars.
  - Level-of-detail selection.
  - Particle effects (dust, plasma, engine trails).
- Keep CPU to GPU synchronization minimal by batching star data into sector buffers.

---

### 1.2 Data Layer (Gaia TAP + Local Evaluation) 📡

This layer retrieves Gaia catalog tuples, converts them into 3D positions, and evaluates them for navigation and gameplay.
Gaia provides observational data, not pre-interpreted objects, so the game must derive meaning from the tuples.

**What Gaia provides**

Each Gaia source is a tuple containing fields such as:

- Source ID
- Right ascension (RA)
- Declination (Dec)
- Parallax
- Proper motion
- Photometric magnitudes

These represent points of light observed by the Gaia instruments.
The game must convert these into spatial objects with derived properties.

**How to implement it**

- Use async TAP queries to retrieve sources within a defined region (cone or box).
- Convert RA, Dec, and parallax into Cartesian coordinates:
  - `d = 1 / parallax`
  - `x = d * cos(Dec) * cos(RA)`
  - `y = d * cos(Dec) * sin(RA)`
  - `z = d * sin(Dec)`
- Normalize units (parsecs or light years) and apply a game scale factor.
- Store each processed source as:
  - Raw Gaia tuple
  - Derived 3D position
  - Derived brightness and color
  - Game metadata (hazard score, resource score, navigational relevance)
- Build a spatial index (KD tree, octree, or grid) for fast queries.
- Evaluate sectors along the ship flight path:
  - Density of sources
  - Distribution of magnitudes
  - Proximity to bright or nearby sources
  - Derived hazard and resource metrics

**Local caching**

- Store processed sectors in a local database (SQLite or binary blobs).
- Load sectors on demand as the ship moves.
- Evict distant sectors to control memory usage.

### 1.3 Gameplay Layer (Navigation, Hazards, Systems) 🎮

This layer uses the evaluated Gaia data to drive navigation, hazards, and ship systems.
All gameplay logic is grounded in the catalog data.

**Navigation**

- The ship moves through a continuous 3D space defined by Gaia positions.
- Build a navigation graph:
  - Nodes: selected Gaia sources or synthetic waypoints.
  - Edges: possible jumps constrained by distance and hazard.
- Use A\* or Dijkstra to compute routes.
- Fuel cost and travel time depend on distance and local conditions.

**Hazards and resources**

- Hazard score is derived from:
  - Brightness
  - Distance to the source
  - Local density
- Resource score is derived from:
  - Sector density
  - Game rules for resource distribution
- These scores influence:
  - Damage over time
  - Fuel consumption
  - Availability of refueling or repair opportunities

**Ship systems**

- Systems include engines, shields, sensors, life support, and cargo.
- Each system has:
  - Health
  - Power draw
  - Efficiency modifiers
- Sensors improve the accuracy of sector evaluation.
- Navigation computers improve route planning.

**Event generation**

- Events are conditioned on Gaia-derived context:
  - High hazard sectors produce radiation spikes or system overloads.
  - High resource sectors produce salvage or refuel opportunities.
- This ensures events are grounded in real catalog structure.

## 🌠 2. Using Gaia Data to Build the Game World

### ⭐ 2.1 Convert Gaia DR3/DR4 into a 3D Galaxy

Convert RA/Dec/parallax to Cartesian:

```text
d = 1/parallax
x = d * cos(Dec) * cos(RA)
y = d * cos(Dec) * sin(RA)
z = d * sin(Dec)
```

### 🔭 2.2 Star Classification

Use Gaia fields:

- `teff_val` → temperature
- `phot_bp_mean_mag - phot_rp_mean_mag` → color index
- `lum_val` → luminosity
- `classprob_dsc_*` → star type probabilities

This lets you generate:

- Habitable zones
- Radiation hazards
- Fuel harvesting opportunities
- Rare exotic stars → story events

### 🗺️ 2.3 Sectorization

Divide space into cubes (e.g., 10 ly).

Each sector stores:

- Star list
- Resource density
- Hazard probability
- "Lore seeds" (ancient ruins, derelicts, anomalies)
